# ADR-0004：模块开通与后端访问控制

- 状态：Proposed
- 日期：2026-07-19
- 关联文档：[模块化产品架构规划](../modular-product-architecture.md)

## 背景

菜单隐藏只改善用户体验，不能阻止直接调用 API。模块是否被构建、模块是否可用、租户是否具有合同或套餐权益、租户是否主动启用，以及用户是否拥有操作权限是不同的安全条件，必须由后端统一执行。

当前 RBAC 中超级管理员会绕过权限码校验。该行为不能自然扩展为绕过模块安装、运行状态、租户开通或许可证状态。

## 决策

新增统一访问依赖 `require_module_access(module_code, permission_code)`，业务模块的所有受保护接口都必须使用该依赖。

校验顺序固定为：

```text
模块包含在 Build Manifest
→ desired_state=enabled 且 observed_state=ready
→ 当前租户具有有效模块权益
→ 当前租户未主动停用模块
→ 当前用户拥有权限码
```

模块权益与租户启停必须分开建模：

1. `TenantPlanModule` 定义套餐包含的基础权益。
2. `TenantModuleEntitlementOverride` 由平台管理员维护，可以在合同约定下额外授予或收回权益，并记录原因、操作人和有效期。
3. `TenantModule` 只记录租户在已有权益内的启用偏好，不能创建超出套餐或平台授权的权益。
4. 租户管理员只能修改 `TenantModule`，不能修改套餐权益或 entitlement override。
5. 租户、套餐、权益或模块本身不可用时一律拒绝。
6. 默认拒绝，不能因为菜单、角色或历史授权残留而推断模块已开通。

有效权益按以下顺序计算：

```text
TenantPlanModule
→ 应用有效的 TenantModuleEntitlementOverride
→ 检查 TenantModule 未主动停用
```

超级管理员只绕过最后一层 RBAC 权限检查，不绕过以下条件：

- 模块不在当前构建产物中。
- 模块未完成迁移或健康检查失败。
- 当前租户没有有效权益或已主动停用模块。
- 租户或模块已被停用。

平台级模块管理接口与业务接口分离。平台超级管理员可以通过模块管理接口配置权益覆盖和租户启用偏好，但不能借此直接访问无有效权益或已停用租户的业务数据。

动态菜单和权限列表使用同一套模块访问判定：

```text
已安装且就绪 ∩ 租户有效权益 ∩ 租户启用偏好 ∩ 角色菜单授权
```

模块开通状态改变时必须：

- 递增租户授权版本并使 RBAC/菜单缓存失效。
- 写入操作审计日志。
- 使后续 API 请求立即按新状态判断。
- 通知相关 Worker 停止领取该租户在此模块中的新任务。

统一错误语义：

| 场景 | HTTP 状态 | 错误码 |
| --- | --- | --- |
| 当前发行版未包含模块 | 404 | `MODULE_NOT_INSTALLED` |
| 模块正在迁移或故障 | 503 | `MODULE_UNAVAILABLE` |
| 租户无有效模块权益 | 403 | `TENANT_MODULE_ENTITLEMENT_REQUIRED` |
| 租户主动停用模块 | 403 | `TENANT_MODULE_DISABLED` |
| 用户缺少操作权限 | 403 | `PERMISSION_REQUIRED` |

## 结果

优点：

- 模块授权成为真正的后端安全边界。
- 套餐、租户覆盖和角色权限职责清晰。
- 套餐权益与租户启停分离，租户管理员不能自行突破产品权益边界。
- 超级管理员不会意外绕过商业授权和租户隔离。

代价：

- 每个业务 Router 必须声明模块编码。
- 需要设计缓存版本和跨进程失效机制。
- 定时任务、事件消费者和导出任务也必须复用模块判定。

## 验收标准

- 租户无 ERP 权益时，直接请求 ERP API 返回 `TENANT_MODULE_ENTITLEMENT_REQUIRED`。
- 租户有 ERP 权益但主动停用时，直接请求返回 `TENANT_MODULE_DISABLED`。
- 超级管理员不能直接访问未开通租户的 ERP 业务接口。
- 停用模块后菜单消失，API、Worker 和定时任务均停止接收新工作。
- 重新开通模块后不需要重新分配原有角色权限即可恢复授权，但必须保留审计记录。
