# ADR-0007：平台主数据生命周期与跨模块引用

- 状态：Proposed
- 日期：2026-07-19
- 关联文档：[模块化产品架构规划](../modular-product-architecture.md)

## 背景

ERP、IOA 会引用平台拥有的租户、用户、部门和岗位。架构禁止业务模块直接查询或外键关联其他模块的数据表，因此数据库无法自动阻止平台删除仍被业务单据引用的主数据。

当前系统在用户只剩最后一个租户成员关系时可能物理删除用户。引入 ERP、IOA 后，这会使历史单据、审批实例和审计记录中的 `user_id` 变成无法解析的孤儿引用。

## 决策

平台主数据采用不可变标识、逻辑归档和生命周期事件：

- `tenant_id`、`user_id`、`department_id`、`post_id` 创建后不可修改且永不复用。
- 已被业务模块引用的主数据默认只允许停用或归档，不执行物理删除。
- 业务模块保存稳定 ID，并为历史凭证保存必要快照，例如姓名、部门名称和岗位名称。
- 快照只用于还原历史事实，不作为当前权限、组织关系或身份状态的判断依据。
- 业务模块通过平台服务接口或模块内端口解析当前主数据，不直接读取平台表。

主数据状态至少区分：

```text
active → inactive → archived → anonymized
```

- `inactive`：禁止新的登录、分配或业务选择，但保留现有引用。
- `archived`：从日常列表隐藏，历史记录仍可解析。
- `anonymized`：在法律或隐私要求下清除个人信息，但保留不可复用的匿名标识和必要审计链。

平台通过事务 Outbox 发布版本化生命周期事件：

```text
UserDeactivated
UserArchived
UserAnonymized
DepartmentArchived
PostArchived
TenantArchived
```

消费者必须幂等更新自己的只读投影或停止创建新关联。生命周期事件不授权业务模块修改平台主数据。

### 物理删除协议

物理删除属于例外操作，必须同时满足：

1. 数据保留期、审计要求和法律要求允许删除。
2. 所有已安装模块通过统一 `ReferenceGuard` 报告不存在有效引用。
3. 相关 Outbox 事件已经成功投递。
4. 操作已备份并经过平台级审批和审计。

无法确认某个模块的引用状态时默认拒绝物理删除。未来模块拆为独立服务后，`ReferenceGuard` 通过有超时和失败关闭策略的内部接口实现。

用户提出隐私删除请求时优先执行匿名化，而不是删除历史业务链所依赖的标识。

## 结果

优点：

- 避免没有跨模块外键时产生不可解释的孤儿数据。
- 历史单据既能还原当时事实，又不会误用快照进行当前授权。
- 为用户离职、组织调整、租户归档和隐私处理提供统一规则。

代价：

- 现有用户、部门和岗位删除接口需要逐步改为停用或归档语义。
- 模块需要保存必要快照并消费生命周期事件。
- 物理删除流程需要跨模块引用检查。

## 验收标准

- 归档用户后，历史 ERP 单据仍能显示创建人快照，但该用户不能登录或创建新业务。
- 删除接口不能物理删除仍被任意已安装模块引用的用户、部门或岗位。
- 用户匿名化后不泄露个人信息，同时保留审计和财务记录要求的匿名标识。
- 生命周期事件重复投递不会产生重复状态变化。
- 某个模块的引用检查不可用时，物理删除请求失败关闭。
